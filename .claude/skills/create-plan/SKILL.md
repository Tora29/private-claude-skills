# create-plan

ユーザーのインプット（実装したい機能・変更内容）を受け取り、
Claudeが実装しやすい構造化された計画書を `.claude/plan/plan.md` に生成する。

---

## ステップ 1: インプット確認と調査

ユーザーの発話から以下を読み取る。読み取れない場合は徹底的にユーザーに確認する。

- 何を作りたいか / 何を変えたいか
- 既存コードの変更か・新規機能か
- 制約や前提条件（デッドライン、技術選定の縛りなど）

その後、実装に関係するファイルをリポジトリから調査する。

```bash
# 関連するファイルを探す
find src/routes/ src/lib/ -name "*.ts" -o -name "*.svelte" | grep -v "node_modules" | head -30
find specs/ -name "spec.md" | head -10
git log --oneline -10
```

`AGENTS.md`、`.claude/rules/` のルールも確認し、計画に反映させること。

---

## ステップ 2: 既存 plan.md の確認

`.claude/plan/plan.md` が既に存在する場合、内容を読んでユーザーに提示し、上書きしてよいか確認する。

> `plan.md` が既にあります（タスク名: {既存のタスク名}）。上書きしてよいですか？

承認が得られた場合のみ上書きする。「いいえ」「残して」などの場合は処理を中断する。
存在しない場合はそのまま次のステップへ進む。

---

## ステップ 3: 計画書の生成

調査結果をもとに `.claude/plan/plan.md` を生成する。
`references/template.md` のテンプレートに従って作成する。

### 計画書の品質基準

計画書を読むだけで実装に着手できる状態にする。以下が揃っていること。

| 項目             | 基準                                                                       |
| ---------------- | -------------------------------------------------------------------------- |
| タスク概要       | 「何を・なぜ」が3行以内で分かる                                            |
| スコープ         | やること・やらないことが明示されている                                     |
| 対象ファイル     | 変更するファイルのパスと役割が全て列挙されている                           |
| 実装順序         | 依存関係を考慮した番号付き順序になっている                                 |
| タスク詳細       | 関数シグネチャ・入出力型・エラー方針が書かれている                         |
| テスト方針       | `testing.md` の Vitest/Playwright ルールに沿ったテスト対象と検証内容がある |
| 実装後の期待状態 | システム・ユーザー・運用の変化が変更前後で明示されている                   |
| 完了条件         | チェックリスト形式で確認可能                                               |

### 注意点

- 実装は**しない**。計画書を書くだけ。
- 曖昧な表現（「適切に処理する」など）を使わない。具体的に書く。
- `.claude/rules/` の各ルール（error-handling, schemas, api-patterns, file-headers, testing, svelte, data-testid など）に準拠した実装方針を計画に組み込む。
- D1 マイグレーションが含まれる場合は `src/lib/server/tables.ts` と `migrations/` の構造も確認して計画に含める。

---

## ステップ 4: 確認と調整

計画書を生成したら、ユーザーに以下を伝える。

1. 計画書のパス（`.claude/plan/plan.md`）
2. 計画書の概要（何ファイル・何ステップか）
3. 不明点・決めていない選択肢があれば提示する

ユーザーのフィードバックを受けて計画書を修正する。
「OK」「進めて」「実装して」などの承認が出たら計画書作成完了。

---

## ステップ 5: GitHub issue 起票（確認制）

計画書が確定したら、ユーザーに確認する。

> GitHub issue として起票しますか？

「はい」「起票して」などの承認が得られた場合のみ、以下のコマンドで起票する。

```bash
gh issue create \
  --title "{タスク概要タイトル}" \
  --body "$(cat <<'EOF'
## 概要

{plan.md のタスク概要セクションをそのまま転記}

## スコープ

### やること
{箇条書き}

### やらないこと
{箇条書き}

## 対象ファイル

{ファイル一覧テーブル}

## 完了条件

{チェックリスト}

## 参照

- 詳細設計: `.claude/plan/plan.md`
EOF
)"
```

起票後は issue URL をユーザーに伝える。

「不要」「いい」などの場合はスキップして終了する。
